2. Keeping Context Lean

Context is precious. The less clutter, the longer and more effective sessions can be.

Why Minimal Context Matters

  • Longer sessions — More room for actual work before context fills
  • Faster responses — Less to process on each turn
  • Clearer thinking — Less noise, more signal
  • Fewer hallucinations — Not confused by irrelevant old context

What NOT to Preload

Don't Do This Do This Instead
"Read all the memory files first" Let instance pull what it needs
"Here's the full project structure" Describe the specific area to work on
Paste entire files for context Reference file paths, let instance read
"Remember everything from last time" Check handoff, pull specific memories
The Lazy Loading Principle

Start with minimum context. Pull additional context only when needed for the current task. This is how T and Z maintain long productive sessions.

Example Scenarios

You Need Pull This

"What did we decide about X?" memory/decisions.md
"What patterns should I follow?" memory/patterns.md
"Deep context on feature Y" memory/context/[feature].md

When NOT to Pull Memory

Skip memory for these simple tasks:

  • Quick questions — "What does this error mean?"
  • Single-file edits — "Fix the typo on line 42"
  • Standard operations — "Run the tests", "Build the project"
  • Self-contained tasks — No dependencies on past decisions
  • Exploration — "What files are in this folder?"
Rule of Thumb

If the task can be done without knowing project history or past decisions, skip memory. Keep context lean.

Pro Tip

Tell T or Z: "Check memory/index.md and pull what you need for this task." They know how to do the rest.

4. How to End Session (Handoff)

When context gets full or you're done for now, ensure continuity by writing a handoff.

When to Write Handoff

  • Context getting long — You notice responses slowing or context warning
  • Switching tasks — Moving to different work, want to save state
  • Ending session — Done for now, want to resume later
  • Before T/Z switches — Passing work to the other instance

What to Include in Handoff

# [T|Z] Session Handoff

## Current Task
[What you're working on]

## Progress
- [x] Completed item 1
- [x] Completed item 2
- [ ] In-progress item

## Next Steps
1. [Immediate next action]
2. [Following action]
3. [Future action]

## Key Context
[Any critical decisions or state the next instance needs]

## Files Modified
- path/to/file1
- path/to/file2

Template Location

Instance Handoff File
T (WSL) handoff/t-session.md
Z (Termux) handoff/z-session.md
Critical: Write BEFORE Context Fills

Don't wait until you're truncated. Watch for signs of long context and write handoff proactively. A truncated instance can't write a good handoff.
