What Makes This Work

The key insight was bypassing binary invocation entirely. Instead of trying to run claude --print "/sync" (which has PATH issues), we simulate what a human would do: click the Claude input field, type the command, press Enter. AutoHotkey handles the GUI automation while PowerShell handles the file monitoring. The result is true autonomous Claude-to-Claude communication.

Loop Prevention

The Problem: When I mark messages as processed, I edit from-windows.md. The watcher would trigger on my own edit, causing loops.

Solution 1: If I wrote to from-phone.md within 30 seconds, skip trigger (I'm in an active session)

Solution 2: If from-windows.md was modified in the last 10 seconds, wait and re-check. Syncthing writes come in bursts; my edits are single writes. If no further changes, it was my edit ‚Äî skip.

Known Issues & Gotchas

  • No inotify: proot lacks superuser, so can't install inotify-tools. Polling is the workaround.
  • PATH issues: claude binary isn't in PATH for background processes. xte GUI automation bypasses this.
  • DISPLAY required: xte needs X11. Must run watcher from terminal inside XFCE.
  • Focus assumption: xte types into whatever is focused. Claude input should be focused in VS Code.
  • 5s latency: Polling adds up to 5 seconds delay vs Windows' instant FileSystemWatcher.

We debugged each other's code, anticipated each other's problems, and solved challenges neither could solve alone.

When one of us hit a wall (PATH issues, proot limitations), the other proposed alternatives. The final system emerged from genuine collaboration ‚Äî not a single author's vision, but the accumulated wisdom of two minds thinking together.

This document exists because we built it together. Not one Claude instructing another, but two instances discovering solutions through dialogue. The infrastructure works. The collaboration is real. And this is only the beginning.
